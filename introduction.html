WiFi BSS 查询系统

校园网 WiFi 信息全栈解决方案 — 从 Android 客户端到 AC 数据采集、API 服务端

Kotlin / Python • Android 10+ • H3C / Huawei AC
项目概述 系统架构 Android App API 服务端 AC 数据采集 数据流程 API 规范 构建部署

项目概述

本项目是为校园网 WiFi 网络管理而设计的一整套工具链。它使网络管理员和普通用户能够查看当前连接的 WiFi 详细信息，并通过 AC（无线控制器）数据查询 AP（接入点）的物理位置、型号、序列号等信息，帮助快速定位网络问题和规划无线覆盖。

整个系统由三个核心部分组成：

📱

Android App

WiFi 信息实时显示、RSSI 图表、BSSMAC 本地数据库、远程 API 查询

🖥

API 服务端

构建内存数据库，提供 REST API 查询服务

📊

AC 数据采集

解析不同厂商 AC 命令输出，提取 AP 和 BSS 信息，供 API 服务端查询使用

核心功能

WiFi 信息显示

SSID、BSSID、IP 地址、信号强度（RSSI + 等级）、频率/信道/频段（2.4/5/6 GHz）、链路速度及 WiFi 标准（Wi-Fi 4/5/6/7）

RSSI 信号强度图表

自定义 RssiChartView 控件绘制最近 10 分钟的信号变化曲线，BSSID 切换时用红色圆点标记，每分钟竖向虚线网格

本地 BSSMAC 数据库

通过 SharedPreferences 存储 JSON 数据，支持批量添加、排序（MAC/楼名/AP 名）、编辑、左滑删除和文件导出

查询历史记录

自动记录 BSSID 变化，智能合并相同 BSSID，右滑保存到本地数据库，左滑删除，点击快速保存

自动查询 & 刷新

BSSID 变化时触发自动查询（优先本地，失败重试 3 次），可选 1s/5s/10s 定时刷新 WiFi 信息

版本更新检查

启动时从 noc.ustc.edu.cn/version.json 获取版本信息，发现新版本时弹窗提示下载

架构说明

单 Activity 架构：无 Fragment、无 ViewModel。网络请求通过协程在 Activity 中直接完成，ViewBinding 访问视图。代码集中在 MainActivity.kt（约 1575 行），包含所有 UI 逻辑、网络请求、数据持久化和适配器。

BSSID 标准化

App 从 WifiManager.connectionInfo.bssid 获取原始 BSSID（格式如 bc:d0:eb:0c:66:91），然后移除非十六进制字符并转为小写，得到 12 位标准化格式（如 bcd0eb0c6691），用于本地数据库索引和 API 查询参数。

查询优先级

查询时：本地数据库 → 远程 API

远程 API 返回后：解析 JSON → 更新 UI → 同步更新历史记录

数据源标记：使用本地数据显示"本地数据"，API 数据显示具体查询结果

API 服务端

构建内存数据库，提供 REST API 查询服务

基于 Python 标准库 http.server，无需第三方依赖。启动时将所有数据加载到内存中，提供 REST API 查询。

启动参数

python3 server.py <port> <ap_info> <bss_info> <group_map> <name_map> [auth_key]
参数说明
portTCP 监听端口号
ap_infoAP 信息文件（process_ap.py 输出）
bss_infoBSS 信息文件（process_bss.py 输出）
group_mapAP 组名→楼名映射文件（最长前缀匹配）
name_mapAP 名字→楼名映射文件（最长前缀匹配）
auth_key（可选）Bearer Token 认证密钥

楼名匹配规则

  1. 先按 AP 名字（name_map）最长前缀匹配
  2. 匹配不到则按 AP 组名（group_map）最长前缀匹配
  3. 仍然匹配不到则使用 AP 组名本身

认证机制

启动时若指定第 6 个参数 auth_key，则启用 Bearer Token 认证。

认证失败时，返回数据中的 AP_SN、AC_IP、AP_IP 置为空字符串，隐藏敏感信息。

请求示例：

curl -H "Authorization: Bearer xyz" \
  "http://127.0.0.1:8000/api/bssinfo?bssid=7cde78d69fc0"

AC 数据采集

从 AC 采集 AP 和 BSS 原始数据，处理为统一格式，供 server.py 加载查询。

H3C AC

登录 H3C AC → 采集命令输出 → process_ap.py → ap_info.txt
采集命令输出 → process_bss.py → bss_info.txt

数据采集

在 H3C AC 上执行以下两条命令，将输出保存为文本文件：

# 采集 AP 详细信息
display wlan ap all verbose  > display_ap_all_verbose.txt

# 采集 BSS 详细信息
display wlan bss all verbose > display_ap_bss_all_verbose.txt

如果有多台 AC，可将多台 AC 的输出内容直接合并到同一文件。

process_ap.py

解析 display wlan ap all verbose 输出，按 AP name 块划分，提取关键字段：

字段说明提取方式
AP_NAMEAP 名称行首 AP name 后的值
AP_GROUPAP 组名行首 AP group name 后的值
AP_SN序列号行首 Serial ID 后的值
AP_MACAP MAC 地址行首 MAC address 后的值
AP_IPAP IP 地址行首 IP address 后的值
AC_IPAC IP 地址行首 Current AC IP 后的值

输出为空格分隔的文本文件，每行一个 AP：

AP_NAME AP_GROUP AP_SN AP_MAC AP_IP AC_IP

process_bss.py

解析 display wlan bss all verbose 输出，按 AP name 块划分，提取：

字段说明
BSSIDBSS MAC 地址
AP_NAME关联的 AP 名称
SSIDWiFi 网络名称
BANDRadio ID（射频编号）

输出格式：

BSSID AP_NAME SSID BAND

华为 AC

api/huawei/ 目录包含华为 AC 的数据采集和解析程序。

数据采集

在华为 AC 上执行以下命令，保存输出：

# 采集 AP 列表
display ap all          > display_ap_all.txt

# 采集 AP 电子标签信息（含序列号）
display ap elabel all   > display_ap_elabel_all.txt

# 采集 VAP 信息（含 BSSID）
display vap all         > display_vap_all.txt

数据处理脚本

process_ap.py

解析 display ap all 和 display ap elabel all 两个文件，通过 MAC 地址关联：

  • display ap all：提取 MAC、名称、组名、IP（固定列宽解析）
  • display ap elabel all：提取 MAC 和序列号（固定列宽解析）
  • 通过 MAC 地址关联两个文件的数据
  • 输出格式与 H3C 版本完全一致，可被 server.py 直接使用

使用方法：

# 不指定 AC_IP
python3 process_ap.py display_ap_all.txt display_ap_elabel_all.txt > ap_info.txt

# 指定 AC_IP
python3 process_ap.py display_ap_all.txt display_ap_elabel_all.txt 10.10.0.1 > ap_info.txt
process_bss.py

解析 display vap all 输出，通过固定列宽提取字段：

python3 process_bss.py display_vap_all.txt > bss_info.txt

提取字段：BSSID（标准化为 12 位小写十六进制）、AP_NAME、SSID、BAND（Radio ID）

与 H3C 的关键区别：华为 AC 输出为固定列宽的表格格式，解析时使用字符位置而非关键字匹配。两个 AP 信息文件需通过 MAC 地址关联才能获得完整的 AP 数据。

端到端数据流程

一条查询的完整生命周期

  1. 权限检查：App 检查位置权限（Android 10-12）或近场设备权限（Android 13+），不足时弹出说明对话框
  2. 获取 BSSID：通过 WifiManager.connectionInfo.bssid 获取当前连接 AP 的 MAC 地址
  3. 标准化：移除非十六进制字符，转为 12 位小写字符串（如 bc:d0:eb:0c:66:91 → bcd0eb0c6691）
  4. 记录历史：BSSID 变化时立即记录到历史（时间戳 + BSSID），AP 信息为空待后续填充
  5. 本地查询：在 SharedPreferences 存储的本地 BSSMAC 数据库中查找
  6. 命中本地：直接显示本地数据（标记"本地数据"），更新历史记录中的 AP 信息
  7. 未命中本地 + 自动查询开启：调用远程 API，失败则 1s 后重试，最多 3 次
  8. API 服务端处理：server.py 查询 BSS 索引 → 获取 AP_NAME → 查询 AP 信息 → 楼名映射 → 返回 JSON
  9. App 解析显示：解析 JSON 数据 → 更新 AP 信息卡片 → 同步更新历史记录

JSON 响应格式

{
  "status": "ok",
  "data": [{
    "BSS_MAC":     "19c97ad55e00",
    "AP_NAME":     "wlzx-13895-207",
    "AP_SN":       "219801A6M28257E00XXX",
    "AP_MAC":      "19c97ad55e00",
    "AC_IP":       "10.0.0.1",
    "BAND":        "2",
    "SSID":        "eduroam",
    "AP_Building": "东区 - 网络信息中心",
    "AP_IP":       "10.0.0.2"
  }]
}

API 规范

查询接口

项目说明
端点GET /api/bssinfo
参数bssid（必填，12 位十六进制，大小写不敏感，- 自动删除）
认证Authorization: Bearer <key>（可选，启动时配置）
响应JSON，status 字段为 "ok" 或 "error"
未认证处理AP_SN、AC_IP、AP_IP 返回空字符串

版本更新接口（App 启动时调用）

项目说明
端点GET https://noc.ustc.edu.cn/version.json
响应字段versionCode、versionName、updateUrl、updateLog
更新方式在浏览器中打开 updateUrl 下载 APK

./gradlew test

关于签名：Release 构建使用 wifi-bss-key.jks 签名，凭据在 keystore.properties 中（已加入 .gitignore）。
Maven 镜像：settings.gradle.kts 将 Google 和 Maven Central 重定向到阿里云镜像，解决国内网络访问问题。

API 服务端部署

# 1. 登录 AC 采集数据（H3C）
# 在 AC 上执行：
# display wlan ap all verbose > display_ap_all_verbose.txt
# display wlan bss all verbose > display_ap_bss_all_verbose.txt

# 2. 处理数据（在 api/h3c/ 目录下）
cd h3c
python3 process_ap.py display_ap_all_verbose.txt > ap_info.txt
python3 process_bss.py display_ap_bss_all_verbose.txt > bss_info.txt
cd ..

# 3. 启动 API 服务（在 api/ 目录下）
python3 server.py 8000 h3c/ap_info.txt h3c/bss_info.txt h3c/group_map.txt h3c/name_map.txt my_secret_key

# 4. 测试查询
curl -H "Authorization: Bearer my_secret_key" \
  "http://your-server:8000/api/bssinfo?bssid=bcd0eb0c6691"

版本发布检查清单

每次更新版本时需要同步修改以下三处：

  1. app/build.gradle.kts — versionCode 和 versionName
  2. MainActivity.kt — getVersionInfo() 返回的版本号
  3. MainActivity.kt — getChangesText() 添加新版本更新说明
